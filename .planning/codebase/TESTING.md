# Testing Patterns

**Analysis Date:** 2026-02-12

## Test Framework

**Runner:**
- Node built-in test runner (`node:test`).
- Tests are executed from compiled JS output:
  - Script: `node --test dist/__tests__` in `packages/server/package.json`.

**Assertion Library:**
- `node:assert/strict` (for example `packages/server/src/__tests__/cliTools.test.ts`).

**Run Commands:**
```bash
npm install --prefix packages/server
npm run build --prefix packages/server
npm run lint --prefix packages/server
npm run test --prefix packages/server
npm run integration --prefix packages/server
```

## Test File Organization

**Location:**
- Source tests live in `packages/server/src/__tests__/`.
- Build output tests run from `packages/server/dist/__tests__/` (generated by `npm run build`).

**Naming:**
- `*.test.ts` for most tests.
- `cliTools.nonwrite.test.ts` for the non-write invariant test - `packages/server/src/__tests__/cliTools.nonwrite.test.ts`.

**Structure:**
```
packages/server/src/
  __tests__/
    cliTools.test.ts
    cliTools.nonwrite.test.ts
    projectManifest.test.ts
    schemaResources.test.ts
  integration/
    smoke.ts
```

## Test Structure

**Suite Organization:**
- Uses `describe()` and `it()` from `node:test`.
- Uses `assert.*` from `node:assert/strict`.
- Example pattern: `packages/server/src/__tests__/cliTools.test.ts`.

**Patterns:**
- Temporary filesystem fixtures created per test using `fs.mkdtempSync()` and cleaned up with `fs.rmSync()`.
  - Example: `packages/server/src/__tests__/projectManifest.test.ts`.
- External CLI execution is stubbed with small runner functions rather than using a mocking framework.
  - Example: `makeRunnerStub()` in `packages/server/src/__tests__/cliTools.test.ts`.

## Mocking

**Framework:**
- No Jest/Vitest/Sinon found; mocking is done via hand-written stubs.

**Patterns:**
- Runner stub pattern for CLI tools:
  - `RunnerStub` type and `makeRunnerStub()` in `packages/server/src/__tests__/cliTools.test.ts`.

**What to Mock:**
- External command execution boundaries (Drush/Composer runners).
- Temporary project filesystem shapes instead of relying on real Drupal installs.

**What NOT to Mock:**
- Pure normalization logic (covered via unit tests driven by stubbed CLI outputs).

## Fixtures and Factories

**Test Data:**
- Fixture projects created on the fly in OS temp directories.
  - Example: `createTempProject()` in `packages/server/src/__tests__/cliTools.test.ts`.

## Coverage

**Requirements:**
- No explicit coverage thresholds configured.
- Focus is on:
  - CLI parsing/normalization correctness
  - config/error envelope behavior
  - non-write guarantees

## Test Types

**Unit Tests:**
- Tool parsing and manifest behavior (examples under `packages/server/src/__tests__/`).

**Integration Tests:**
- HTTP transport smoke test starts server on ephemeral port and calls endpoints.
  - File: `packages/server/src/integration/smoke.ts`
  - Script: `npm run integration --prefix packages/server` runs `node dist/integration/smoke.js`

**E2E Tests:**
- None.
